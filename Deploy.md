## 🚀 Руководство по развертыванию приложения Family Task Manager (MVP)
Данное приложение спроектировано в рамках Decoupled (Stateless) архитектуры . Бэкенд и фронтенд полностью изолированы и упакованы в раздельные Docker-контейнеры . Оркестрация всей системы (включая СУБД) осуществляется через Docker Compose .
------------------------------
## 🛠 Архитектурные компоненты системы

   1. База данных (postgres_db): Изолированный контейнер postgres:15-alpine. Данные сохраняются на хост-машину через volume pgdata.
   2. Бэкенд-сервер API (task_server): Сервер на Go (Golang) (маршрутизация на chi, авторизация на JWT, валидация через go-playground/validator) \n. Работает на порту 8080.
   3. Фронтенд-сервер (frontend): Статические файлы (HTML5 / CSS3 / Vanilla JS), упакованные в легковесный контейнер Nginx alpine . Работает на порту 3000 .

------------------------------
## 📋 Подготовка к деплою на сервере (Хосте)
Перед запуском убедитесь, что на целевом сервере (Ubuntu/Debian VPS) установлены:

* Docker Engine (версии 20.10+)
* Docker Compose (версии 2.0+)

## 1. Клонирование репозитория
Склонируйте актуальную ветку проекта в рабочую директорию на сервере:

git clone -b feature/refactor-and-subtasks https://github.com /var/www/family-tasks
cd /var/www/family-tasks

## 2. Накатывание миграций БД (Если разворачивается впервые)
Скрипт первоначальной структуры таблиц с поддержкой каскадного удаления подзадач (ON DELETE CASCADE) находится по адресу:
./migrations/000001_init.up.sql
Примечание: Если на сервере база запускается «с нуля» через Docker Compose, Postgres автоматически инициализирует пустую схему.

## 🐳 Сценарий 1: Промышленный деплой «Всё в Docker» (Рекомендуемый)
При этом подходе вся инфраструктура (Бэк, База и Фронт) поднимается одной командой. Вся конфигурация окружения зашита в файле docker-compose.yaml.
## Шаг 1: Конфигурация переменных окружения
Откройте docker-compose.yaml в корне проекта и при необходимости скорректируйте production-значения в секции environment для task_server:

* JWT_SECRET: Установите надежный криптографический ключ сессии.
* REGISTRATION_INVITE_CODE: Секретный инвайт-код для регистрации вашей семьи.

## Шаг 2: Сборка и запуск оркестратора
Выполните команду принудительной сборки образов из исходников на сервере:

docker compose up -d --build

## Шаг 3: Проверка статуса сервисов
Убедитесь, что все три контейнера перешли в статус Up:

docker compose ps

------------------------------
## 🛠 Сценарий 2: Раздельный деплой (Классический PHP-стиль)
Если вы хотите раздавать фронтенд силами системного Nginx хост-машины (вместо контейнера фронтенда), используйте этот сценарий.
## Шаг 1: Запуск только Бэкенда и Базы
Измените файл docker-compose.yaml, удалив или закомментировав секцию сервиса frontend. Запустите только серверную часть:

docker compose up -d --build postgres_db task_server

В результате Go API будет доступен по адресу http://localhost:8080/api/v1.
## Шаг 2: Деплой статики Фронтенда на хост

   1. Скопируйте файлы из папки ./web в корневую директорию веб-сервера на хосте (например, в /var/www/family-tasks/frontend):
   
   mkdir -p /var/www/family-tasks/frontend
   cp -r ./web/* /var/www/family-tasks/frontend/
   
   2. Настройте конфигурационный файл системного Nginx сервера (/etc/nginx/sites-available/family-tasks) для раздачи статики и проксирования запросов на Go-API, чтобы избежать проблем с CORS-политикой.

## 📝 Спецификация API (Контракты обмена данными)
Все эндпоинты бэкенда ожидают заголовки 'Content-Type: application/json'.

   1. POST /api/v1/auth/register — Регистрация. Тело: {"username": "Имя", "password": "...", "invite_code": "..."}.
   2. POST /api/v1/auth/login — Аутентификация. Тело: {"username": "Имя", "password": "..."}. Возвращает: {"token": "JWT"}.
   3. GET /api/v1/tasks — Получение общей базы задач семьи. Требует заголовок Authorization: Bearer JWT. Возвращает структуру задач со вложенным массивом subtasks.
   4. PUT /api/v1/tasks/{id} — Редактирование параметров большой задачи. Тело: {"title": "...", "priority": "...", "done": false}.
   5. PUT /api/v1/tasks/subtasks/{sub_id} — Интерактивное переключение статуса пункта чек-листа. Тело: {"done": true/false}. Возвращает 204 No Content.
